# Multivac Product Requirements Document（PRD）

## 1. Introduction

Multivac 是一个以 GTD 方法为基础、结合 AI 能力的高效时间管理工具，帮助用户更好的专注于当前任务，使用户掌控生活、减少焦虑并且提高效率。

### 1.1 Problem Statement

用户在管理多个项目时，会遇到以下问题：
- 任务数量增加，导致用户无法专注于当前任务。
- 任务优先级不清晰，导致用户无法有效管理时间。
- 任务完成时间不准确，导致用户无法及时完成任务。
- 任务缺乏清晰的目标和计划，导致用户无法有效地完成任务。

### 1.2 Solution

Multivac 通过以下方式解决以上问题：
- 提供 GTD 方法的任务管理功能，帮助用户专注于当前任务。
- 结合 AI 能力，自动分析用户任务，提供任务优先级建议。
- 提供任务完成时间预测功能，帮助用户及时完成任务。
- 提供任务目标和计划管理功能，帮助用户清晰定义任务目标和计划。

## 2. User Stories

### 2.1 User Story 1（Capture）

1. 将需要处理的事项从大脑中提取出来，包括工作、生活、灵感等，记录在收集箱（Inbox）中
2. 关键原则：事无巨细、快速记录、集中存放，避免任务遗漏

### 2.2 User Story 2（Clarify）

1. 定期审视收集箱中的事项，明确每个事项的具体内容和意义
2. 关键原则：保持事项的清晰度和可操作性，避免任务模糊；2 分钟原则，如果事项在 2 分钟之内可以完成，就直接完成。
3. 按照 GTD 方法的步骤，进行如下处理
  3.1 如果事项是可立即执行的，放入下一步清单（Next Action）
  3.2 如果事项是委托他人的，放入等待清单（Waiting）
  3.3 如果事项是需要多个步骤才能完成的，放入项目清单（Project）
  3.4 如果事项现在不需要什么动作，则放入将来清单（Later）
  3.5 如果事项是有明确的时间的（即只能在某个时间做的，例如参加会议），放入日程表
  3.6 如果事项不是可执行的动作，仅仅是一些资料，那么放入资料清单（Reference）
  3.7 如果事项是不需要的东西，那么则直接删除

### 2.3 User Story 3（Engage）

根据当前的情境、时间、精力和优先级等，从下一步清单中选择事项去执行。进行如下处理：
1. 如果事项可以委托他人，则放入 Waiting 清单
2. 如果事项需要在某个特定的时间做，那么放入日程表
3. 否则，则直接执行


### 2.4 User Story 4（Reflect）

1. 至少每周回顾一次
2. 回顾的内容，主要是清空收集箱，检查所有清单，确保系统更新

## 3. Product Features

### 3.1 添加事项到 Inbox

1. 用户可以将任何需要后续处理的事项快速添加到 Inbox 中，包括工作任务、生活事务、提醒、灵感和临时想法等。
2. 添加事项时应优先强调快速收集，用户只需使用简短自然语言记录内容，而不需要在录入时立即完成分类。
3. 新增事项默认进入 Inbox，作为后续 Clarify 的统一入口。
4. 产品应支持用户连续添加多条事项，降低 Capture 阶段的记录成本，避免因为录入门槛过高而遗漏事项。

### 3.2 澄清 Inbox 中的事项

1. 用户可以对 Inbox 中的事项逐条进行 Clarify，明确该事项的具体含义以及后续应采取的处理方式。
2. Clarify 的目标是明确事项的去向和下一步处理方式，而不是要求用户在澄清阶段完成所有事项。
3. 对于可立即执行的事项，用户可以将其放入 Next Action。
4. 对于委托他人的事项，用户可以将其放入 Waiting。
5. 对于需要多个步骤才能完成的事项，用户可以将其放入 Project。
6. 对于当前不需要动作但未来可能需要处理的事项，用户可以将其放入 Later。
7. 对于只能在某个明确时间处理的事项，用户可以将其放入日程表。
8. 对于不是可执行动作、仅用于保留和查阅的信息，用户可以将其放入 Reference。
9. 对于不再需要处理的事项，用户可以直接删除。
10. 如果某个事项可以在 2 分钟之内完成，用户可以直接完成该事项，而无需继续保留在系统中。
11. 完成 Clarify 后，该事项不应继续停留在 Inbox 中。

### 3.3 从 Next Action 中选择事项处理

1. Next Action 是用户当前执行事项的主要入口，用户可以从中选择当前最适合处理的一项事项。
2. 用户在选择事项时，可以结合当前的情境、可用时间、当前精力和优先级进行判断。
3. 当用户开始处理某个 Next Action 时，如果该事项更适合委托他人处理，则可以将其转入 Waiting。
4. 如果该事项需要在某个明确的时间处理，则可以将其转入日程表。
5. 否则，用户可以直接执行该事项。
6. 产品应帮助用户将注意力集中在当前真正可执行的事项上，避免在执行阶段再次面对大量模糊任务。

### 3.4 将 Project 拆解为 Actions

1. 对于放入 Project 的事项，用户可以进一步将其拆解为多个具体的 Actions，以便将抽象目标转化为可以执行的步骤。
2. 拆解出的 Actions 应足够具体、清晰且可执行，便于用户理解项目的推进路径。
3. 拆解出的 Actions 仍然属于对应的 Project，用于支持项目的持续推进。
4. 在完成拆解后，是否将某个 Action 放入 Next Action，必须由用户手动选择。
5. 系统不应默认将拆解出的所有 Actions 自动加入 Next Action，以避免 Next Action 清单被大量步骤淹没。
6. 未被用户选入 Next Action 的 Actions，应继续保留在 Project 中，作为该项目后续可推进的步骤。
7. 通过这种方式，用户可以主动控制当前真正进入执行阶段的事项，同时保持 Project 的完整拆解结构。

## 4. Data Model

### 4.1 Inbox

Inbox 是收集箱，用户首先将事项添加到收集箱中，作为后续 Clarify 的统一入口。其中的数据结构如下：

```json
{
    "id": "",   // 事项的唯一标识符, 需要采用一个全局唯一的随机字符串，使用 ulid 生成
    "title": "", // 事项的标题，用户在添加事项时输入的自然语言描述  
    "description": "", // 事项的详细描述，用户在添加事项时输入的详细信息，需要支持 markdown 格式，支持用户通过富文本的方式来编辑
    "created_at": "", // 事项创建的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
    "updated_at": "" // 事项最后更新的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
}
```

### 4.2 Project

Project 是项目清单，用户可以将需要多个步骤才能完成的事项放入 Project 中，以便将抽象目标转化为可以执行的步骤。其中的数据结构如下：

```json
{
    "id": "",    // 项目的唯一标识符, 需要采用一个全局唯一的随机字符串，等于 Inbox.id
    "title": "", // 项目的标题，用户在添加项目时输入的自然语言描述，默认等于 Inbox.title，用户可以修改  
    "description": "", // 项目的详细描述，用户在添加项目时输入的详细信息，需要支持 markdown 格式，支持用户通过富文本的方式来编辑
    "goals": [
        {
            "title": "", // 目标标题，用户在添加项目时输入的目标描述
            "created_at": "", // 目标创建的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
            "completed_at": "" // 目标完成的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区，值为空表示未完成
        }
    ],   // 项目的目标，用户可以在添加项目时输入多个目标
    "references": [
        {
            "title": "", // 参考标题，用户在添加项目时输入的参考描述
            "url": "" // 参考链接，用户在添加项目时输入的参考链接，需要符合 URL 格式
        }
    ], // 项目的参考资料，用户可以在添加项目时输入多个参考
    "status": "Draft", // 项目的状态，默认值为 Draft，其他值为 Active、Completed、Hold，状态机为 Draft -> Active -> Completed，Active -> Hold，Draft -> Hold，Hold -> Active
    "created_at": "", // 项目创建的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
    "updated_at": "", // 项目最后更新的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
    "start_at": "", // 项目开始的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区，为空表示未开始
    "completed_at": "" // 项目完成的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区，值为空表示未完成
}
```

### 4.3 Action

Action 是项目中的一个具体步骤，用户可以在项目中添加多个 Action，一个 Action 也可以不关联任何的项目。其中的数据结构如下：

```json
{
    "id": "",          // Action 的唯一标识符, 需要采用一个全局唯一的随机字符串，如果来自于 Inbox 则等于 Inbox.id，否则使用 ulid 生成
    "title": "",       // Action 的标题，用户在添加 Action 时输入的自然语言描述，如果来自于 Inbox 则等于 Inbox.title，用户可以修改
    "description": "", // Action 的详细描述，用户在添加 Action 时输入的详细信息，需要支持 markdown 格式，支持用户通过富文本的方式来编辑
    "project_id": "", // Action 所属的项目 ID，如果归属于某个 project 则等于 project.id，否则为空
    "kind": "Task",   // 动作的类型，默认等于 Task（下一步任务），其他取值包括 Waiting（等待）、Scheduled（日程）
    "context": [""],  // 一系列关联的情境的 id，用户需要先在 Context 中添加情境，才能将情境关联到 Action 中
    "labels": [
        {
            "name": "" // 标签名称，用户在添加 Action 时输入的标签描述
        }
    ],
    "attributes": {
        "task": {
            "expected_at": "", // 任务的预计完成时间，格式为 RFC33339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
            "status": "Pending" // 任务状态，取值包括 Pending（待处理）、Active（进行中）和 Completed（已完成），默认值为 Pending
        }, // 当 kind 为 Task 时，包含任务相关的属性
        "waiting": {
            "delegatee": "", // 等待的委托人，用户在添加 Action 时输入的委托人
            "due_at": "", // 等待的截止时间，格式为 RFC33339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
        }, // 当 kind 为 Waiting 时，包含等待相关的属性
        "scheduled": {
            "start_at": "", // 日程的开始时间，格式为 RFC33339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
            "end_at": "", // 日程的结束时间，格式为 RFC33339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
        }, // 当 kind 为 Scheduled 时，包含日程相关的属性
    },
    "created_at": "", // Action 创建的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
    "updated_at": "" // Action 最后更新的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
}
```

```json

    "created_at": "", // 情境创建的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区

}
```

### 4.7 Events

Event 是系统中的事件日志，用于记录用户在系统中的操作行为，以便后续进行分析和优化。其中的数据结构如下：

```json
{
    "id": "",          // Event 的唯一标识符, 需要采用一个全局唯一的随机字符串，使用 ulid 生成
    "type": "",        // Event 的类型，用户在系统中执行的具体操作，例如 "CreateInboxItem"、"UpdateProject" 等
    "object": "",      // Event 的对象，用户在系统中执行的具体操作的目标对象 id，例如 InboxItem.id、Project.id 等
    "payload": {},     // Event 的负载，包含与该事件相关的具体数据，例如创建的事项内容、更新的项目信息等
    "created_at": "" // Event 创建的时间，格式为 RFC3339 格式，例如 "2006-01-02T15:04:05Z07:00"，统一使用 UTC 时区
}
```
